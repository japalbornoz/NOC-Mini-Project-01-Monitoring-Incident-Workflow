# Escalation Workflow

## 1. Overview

This document summarizes the incident escalation workflow used in this lab to handle monitoring alerts generated by PRTG and converted into Jira Service Management requests.

The workflow reflects a junior NOC-style response process:
- detect the issue
- validate the alert
- determine impact and severity
- document findings
- escalate when required
- confirm recovery
- update and close the ticket

This is a **lab-based simulated workflow**, not a production support process. It is intended to demonstrate operational thinking, alert handling, and ticket discipline in a small monitoring environment.

---

## 2. Objective

The objective of this workflow is to show how a monitored network issue can move from:

**PRTG alert detection**  
to  
**Jira ticket creation**  
to  
**initial triage and escalation decision**  
to  
**recovery validation and closure**

---

## 3. Scope

This workflow applies to the validated incidents in this mini-project:

- **INC-01 – R1 Router Down**
- **INC-02 – R1 to DSW1 Interface / Link Down**

The workflow is based on:
- PRTG sensor monitoring
- Jira Service Management email intake
- manual L1-style validation steps
- lab recovery and post-check verification

---

## 4. Escalation Model Used in This Lab

Because this project is a solo lab, escalation was simulated using role-based workflow logic rather than actual handoff to another engineer or team.

### Simulated Roles
- **Monitoring System:** PRTG
- **Ticket Intake System:** Jira Service Management
- **L1 / Initial Responder:** Lab operator
- **L2 / Escalation Target:** Simulated higher support tier for issues requiring deeper investigation or corrective action

In a real environment, L2 escalation could mean:
- network engineering
- infrastructure operations
- field support
- vendor support

## 5. Workflow Summary

| Step | Action | Purpose |
|------|--------|---------|
| 1 | PRTG detects sensor failure | Identify abnormal state |
| 2 | Jira request is created from alert email | Record incident automatically |
| 3 | L1 reviews ticket and affected sensor/device | Confirm alert details |
| 4 | L1 validates impact using PRTG and topology awareness | Distinguish outage type |
| 5 | L1 assigns severity and documents findings | Support escalation decision |
| 6 | L1 escalates if recovery is not immediate or root cause requires deeper action | Simulate handoff to higher support tier |
| 7 | Service/device/link is restored | Recover connectivity |
| 8 | PRTG returns to normal state | Confirm restoration |
| 9 | Ticket is updated with findings and resolution | Preserve incident record |
| 10 | Ticket is closed | Complete workflow |

## 6. Detailed Escalation Workflow

### Step 1 – Alert Detection
PRTG detects a sensor entering a failed state such as:
- Down
- Unreachable
- timeout or no response condition

For this lab, alerts were generated from monitored Ping v2 and related device sensors.

### Step 2 – Automatic Ticket Creation
The configured PRTG notification trigger sends an email to Jira Service Management.

This produces a Jira request containing:
- alert subject
- affected device
- affected sensor
- status
- error message
- lab environment tag

### Step 3 – Initial Ticket Review
The responder reviews the newly created Jira request and confirms:
- which device is affected
- which sensor triggered the alert
- whether the ticket content matches the PRTG event
- whether the issue appears to be device-wide or path-specific

### Step 4 – Initial Triage
The responder checks the monitoring view and topology context to determine the likely incident type.

Examples:
- **R1 down with multiple R1 sensors failing** suggests a **device outage**
- **R1 remains healthy while DSW1 becomes unreachable** suggests a **downstream path or interface issue**

### Step 5 – Severity Assignment
Severity is determined based on observed scope and impact.

| Severity | Typical Condition in This Lab | Example |
|----------|-------------------------------|---------|
| High | Core router/device outage affecting monitored path and downstream visibility | INC-01 – R1 Router Down |
| Medium | Partial path or interface failure affecting downstream device reachability while upstream device remains available | INC-02 – R1 to DSW1 Interface / Link Down |

### Step 6 – Escalation Decision
Escalation is required when:
- the issue is not immediately recoverable
- the fault affects critical connectivity
- the responder can verify the symptom but cannot fully remediate the cause
- the event needs deeper infrastructure troubleshooting

In this lab, escalation was simulated after alert validation and severity classification.

### Step 7 – Recovery
Corrective action is performed in the lab by restoring the affected node or interface.

Examples:
- restarting **R1** for router outage validation
- bringing **DSW1 e0/0** back up for interface/link failure validation

### Step 8 – Post-Recovery Validation
Recovery is confirmed by checking:
- PRTG sensor return to Up/OK state
- device visibility restoration
- expected connectivity restoration in the lab

### Step 9 – Ticket Update
The incident record is updated with:
- what failed
- how it was detected
- what impact was observed
- what corrective action restored service
- how recovery was validated

### Step 10 – Closure
The ticket is considered ready for closure when:
- the condition is resolved
- monitoring returns to normal
- validation is complete
- incident notes are documented

## 7. Escalation Logic by Incident Type

### Incident 01 – R1 Router Down
**Escalation path:** High severity

Reason:
- R1 is the monitored upstream router
- multiple R1 sensors fail together
- downstream visibility is affected
- this indicates a broader device/service outage

Expected L1 action:
- confirm alert
- verify device-wide failure pattern
- open/update ticket
- escalate to higher support tier if not immediately restored

### Incident 02 – R1 to DSW1 Interface / Link Down
**Escalation path:** Medium severity

Reason:
- R1 remains reachable
- DSW1 becomes affected downstream
- issue points to a path/interface problem rather than total upstream device loss

Expected L1 action:
- confirm upstream device is still healthy
- identify downstream impact scope
- document likely path failure
- escalate if interface restoration or deeper troubleshooting is required

## 10. Lab Limitations

This workflow was validated in a lab environment with simplified assumptions.

Limitations include:
- no real multi-user support team
- no live SLA or business impact matrix
- no actual on-call escalation chain
- no production communication bridge or outage broadcast workflow

Even with those limits, the workflow still demonstrates the core operational sequence expected in junior monitoring and incident-handling roles.

---

## 11. Conclusion

This lab validated a practical escalation workflow in which:

- PRTG detected the issue
- Jira created the request automatically
- the incident was reviewed and classified
- escalation logic was applied based on scope and severity
- recovery was confirmed through monitoring state restoration

The result is a small but realistic example of how monitoring, ticketing, triage, escalation, and closure fit together in a NOC-style process.
